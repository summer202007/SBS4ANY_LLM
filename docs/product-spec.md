# Product Spec

## Product Thesis

SBS 4 Any Agent is a lightweight local evaluation workbench for agent PMs. It helps a PM turn a task space into a structured eval set, collect side-by-side outputs from a challenger and a ceiling product, then produce an auditable report.

The MVP is not a full benchmark platform. It is a portfolio-grade workbench that demonstrates how an agent PM thinks about task-space-specific evaluation.

## MVP Positioning

The MVP focuses on chatbot SBS evaluation.

- Baseline: Doubao.
- Challenger: any chatbot or agent product whose output can be manually pasted.
- Eval set: generated as draft by Local Codex or GPT API, then edited and approved by a human.
- Collection: manual paste for both sides.
- Report: downloadable local artifact.

The first frontend build should be fixture-first. Use the existing restaurant recommendation runtime eval package as the first real product fixture, then add generation later:

- `artifacts/eval-generation/20260607-restaurant-diandian-vs-doubao/revised-package.json`

This lets the app UI be shaped by the actual eval-generation engine output instead of a simplified early mock model.

## Surface Decision

Build a local desktop Web App, not a mobile app, for MVP.

Reasons:

- package review requires dense structured fields, tables, and trace panels;
- manual SBS collection requires copying prompts and pasting long model outputs;
- side-by-side review is naturally desktop-width;
- local files and Local Codex/GPT provider integration fit desktop better;
- mobile can be deferred to a future read-only/report-review surface.

## Non-Goals

- No browser automation in MVP.
- No account system.
- No cloud backend.
- No shared team workspace.
- No historical run dashboard.
- No universal model marketplace.
- No coding-agent arena in MVP.
- No final grader methodology until the separate evaluation-design session is complete.

## Main User Flow

1. Create Arena
   - Enter task space.
   - Enter challenger name.
   - Baseline is Doubao by default.
   - Choose whether generation can use project context.

2. Load Or Generate Eval Set Draft
   - MVP first loads a validated runtime eval package fixture.
   - Choose Local Codex provider by default, or GPT API key provider.
   - Enter generation settings such as number of cases and single-turn vs multi-turn mix.
   - Generate structured draft cases.

3. Curate Eval Set
   - Review generated cases.
   - Edit case details.
   - Approve or reject cases.
   - Only approved cases enter collection.

4. Collect Outputs
   - For each approved case, show `Case X / Turn Y of Z`.
   - User copies the current turn prompt into Doubao and challenger.
   - User pastes outputs and optional visible process notes back into the app.
   - Store evidence level per side.

5. Review SBS
   - Show Doubao and challenger outputs side by side.
   - Show turn-level and case-level notes.
   - Grader UI is placeholder until evaluation methodology is finalized.

6. Download Report
   - Generate Markdown and/or HTML report.
   - Include arena metadata, eval cases, collected outputs, evidence levels, and review notes.

## Core Screens

### Setup

Fields:

- Arena name.
- Task space.
- Challenger name.
- Baseline name, default `Doubao`.
- Generation provider: `local_codex` or `gpt_api`.
- Use project context: optional.

### Eval Builder

Capabilities:

- Generate draft cases.
- Import JSON later if time permits.
- Edit structured case fields.
- Approve or reject cases.

### Collect

Capabilities:

- Show one case/turn at a time.
- Copy prompt.
- Paste Doubao output.
- Paste challenger output.
- Add visible process notes.
- Set evidence level.

### Review

Capabilities:

- Side-by-side view.
- Case and turn navigation.
- Notes and manual observations.
- Placeholder for future grader outputs.

### Report

Capabilities:

- Preview report.
- Download Markdown or HTML.

## Success Criteria

The MVP is successful if, within one local session, the user can:

1. Generate at least five structured eval cases.
2. Approve a subset of cases.
3. Collect outputs for Doubao and one challenger.
4. Review outputs side by side.
5. Download a report that can be shown as a portfolio artifact.

## Portfolio Success Criteria

The MVP should also produce interview-ready evidence:

1. A runnable local demo.
2. A generated sample eval set that shows task-space thinking.
3. A sample SBS report generated by the app.
4. A README that explains why the product matters and how to run it.
5. A short resume/interview story connecting the app to Agent PM competence.

See `docs/portfolio/portfolio-deliverables.md`.
