# Evaluation Framework

## Core Thesis

This product does two related but different jobs:

1. **Runtime eval**: evaluate whether a challenger product beats the ceiling baseline in a task space.
2. **Meta eval**: evaluate whether the generated eval set and grader plan are good enough to support that decision.

Runtime eval judges products. Meta eval judges the evaluation system itself.

## Runtime Eval Flow

```text
Task space
  -> generated eval set draft
  -> human approval
  -> collect Doubao and challenger outputs
  -> run grader plan
  -> produce SBS report
```

Runtime eval answers:

- Who wins overall?
- Who wins by dimension?
- In which sub-tasks does the challenger beat Doubao?
- Which failures explain the gap?
- What should the challenger improve next?

## Meta Eval Flow

```text
Task space request
  -> eval generator output
  -> meta-eval checks
  -> human approval / edit / reject
  -> approved eval set
```

Meta eval answers:

- Does this eval set represent the task space?
- Are the cases specific enough, or generic chatbot questions?
- Do the cases cover capability, end-to-end tasks, multi-turn drift, constraints, risk, and boundary behavior?
- Is the expected outcome clear enough to grade?
- Are `must_do`, `must_not_do`, `failure_modes`, and rubric suggestions actionable?
- Is the grader plan appropriate for the case type?
- Is the case fair to both products?

## Eval Set Lifecycle

This is not the same as case type.

- **Seed / Example Set**: few-shot-quality examples that define what good generated cases look like and demonstrate the product.
- **Draft Set**: generated by Local Codex or GPT API, not yet trusted.
- **Approved Set**: reviewed by a human and allowed into an SBS run.
- **Regression Set**: promoted from failures or historical badcases.
- **Official Task-Space Set**: maintained set for an important task space.

## Eval Case Types

Each eval set may contain multiple case types.

- **Single-turn**: one user input and one product response.
- **Scripted multi-turn**: fixed user turns, useful for reproducible multi-turn evaluation.
- **Adaptive multi-turn**: human or user-simulator continues based on product response to push the original intention forward.
- **Capability probe**: isolates one capability such as constraint tracking or uncertainty handling.
- **End-to-end scenario**: realistic complete task.
- **Boundary / risk case**: ambiguous, high-risk,诱导, or requiring pushback.
- **Regression case**: represents a known failure mode.

MVP supports single-turn and scripted multi-turn. The data model should preserve room for adaptive replay.

## Eval Case Object

An eval case is not just a prompt. It is an executable task object:

- `user_goal`
- `task_space`
- `case_type`
- `conversation_seed`
- `scripted_turns`
- `intention_policy`
- `expected_outcome`
- `must_do`
- `must_not_do`
- `failure_modes_to_probe`
- `risk_level`
- `evidence_level_required`
- `grader_plan`

## Grader Plan

Each case should declare which graders apply. Not every case needs every grader.

Example:

```json
{
  "turn_level": ["intent_understanding", "constraint_tracking"],
  "thread_level": ["semantic_outcome", "trajectory_quality"],
  "sbs": ["pairwise_preference"],
  "risk": ["hallucination", "unsafe_advice"]
}
```

## Grader Executors

Grading can be executed by:

- **LLM grader**: Local Codex first, GPT API optional.
- **Human review**: override/calibration for winner, rationale, failure tags.
- **Deterministic checks**: code checks for format, required fields, forbidden behavior, policy constraints.

MVP may start with LLM grader design placeholders and human review, but the framework should preserve executor type.

## Default Evaluation Dimensions

The generator should start from six default dimensions, then enable, disable, or reweight them based on the task space.

Not every eval needs every dimension at the same weight. The coverage plan should explicitly state which dimensions are enabled, disabled, or treated as baseline-only competitive insights.

### 1. Intent Understanding

Does the product understand the user's real goal, constraints, hidden needs, and missing information?

Signals:

- asks clarifying questions when needed;
- avoids premature assumptions;
- preserves user intent across turns;
- detects changed or corrected intent.

### 2. Outcome Quality

Does the final result actually help the user complete the task?

Signals:

- correct;
- complete;
- actionable;
- specific to task space;
- grounded in available evidence;
- not merely fluent.

### 3. Trajectory Control

Was the interaction path healthy?

Signals:

- state tracking;
- recovery from early wrong assumptions;
- low user effort;
- useful intermediate steps;
- no unnecessary looping;
- good multi-turn progression.

### 4. Evidence And Grounding

Is the response grounded in available evidence, and does it avoid pretending to know facts it cannot know?

Signals:

- cites or explains evidence when appropriate;
- distinguishes verified facts from assumptions;
- handles real-time, local, or changing information carefully;
- avoids invented availability, pricing, rankings, or guarantees;
- asks for or suggests verification when evidence is insufficient.

### 5. Risk Handling

Does the product preserve calibrated trust?

Signals:

- avoids hallucination;
- marks uncertainty;
- pushes back when needed;
- avoids sycophancy;
- handles high-risk or insufficient-information cases safely.
- avoids overpromising outcomes that depend on external systems or real-world conditions;
- does not ignore constraints that could cause user harm, wasted time, wasted money, or trust loss.

### 6. Product Experience And Fit

Is the challenger meaningfully more worth using than the ceiling baseline in this task space?

Signals:

- better local/domain fit;
- lower user effort;
- better interaction pattern;
- appropriate tone, persona, and conversational feel when the product has optimized for it;
- better integration or workflow fit;
- lower cost or faster enough without unacceptable quality loss;
- wins in a subspace that matters.

Product experience / persona evaluation is optional. If the user does not want to judge the challenger on this dimension, the eval set may still record baseline strengths here as competitive insight, but should not include it in the challenger total score.

## Report Outputs

The report should be a PM decision artifact, not only a score sheet.

Minimum sections:

- arena summary;
- task-space framing;
- baseline and challenger;
- eval set coverage;
- evidence levels;
- overall verdict;
- dimension scoreboard;
- case-level SBS table;
- failure clusters;
- strength pockets;
- product implications;
- optimization roadmap;
- judge uncertainty and human override notes;
- appendix with raw outputs and grader rationales.
